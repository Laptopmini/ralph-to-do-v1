# CLAUDE.md

## Project Overview

ralph-node is a template repository implementing a Maestro-orchestrated, test-gated AI development cycle for Node.js projects. A human submits a feature request; **Maestro** drives the entire lifecycle — generating an implementation blueprint, breaking it into tickets, producing failing tests (backpressure), and running test-gated implementation loops — while gating each phase behind a human-reviewed pull request. **Ralph** is the inner test-gated implementation loop; it is one phase inside Maestro, not the entry point.

This repo contains only the loop infrastructure. Application code is generated by AI agents executing PRDs.

## Bootstrap

The project must be initialized before first use:

```bash
nvm use           # Node v24.14.1 (see .nvmrc)
bash init.sh      # Creates package.json, installs deps, moves the init PRD to PRD.md,
                  # runs a single Ralph loop to bootstrap, then self-destructs
```

After init, `npm run maestro` is the primary entry point for new feature development.

## Commands

- `npm run maestro` — Top-level orchestrator: blueprint → tickets → backpressure → implementation, with PR gates
- `npm run ralph` — Single test-gated implementation loop for one PRD (used internally by Maestro; takes an archive folder argument)
- `npm run backpressure` — Generate failing tests for the active `PRD.md`
- `npm test` — Run Jest unit tests then Playwright E2E tests (sequential)
- `npm run lint` — Lint & format with Biome (auto-fix)
- `npm run check-types` — `tsc --noEmit`

## Code Style

- Formatter/linter: Biome (see `biome.json`)
- 2-space indentation, 100-char line width
- Double quotes, trailing commas, always semicolons
- `app/` and `pages/` directories: default exports are allowed (Next.js convention)

## Testing

- **Unit tests:** Jest — `.test.ts` files for logic-only tasks
- **E2E tests:** Playwright — `.spec.ts` files for UI tasks
- Use `data-testid` attributes as element selectors. Do not use class names or CSS selectors.
- Every test must fail against a blank/missing implementation. No trivially passing assertions.
- Minimum coverage per test file: happy path + most likely failure/edge case

## Maestro Architecture

Maestro orchestrates feature development as a multi-phase pipeline. A human reviews and merges a pull request between each phase, so an entire feature can be supervised purely through PRs.

### Phases

1. **Blueprint** — The `/blueprint` skill (Opus) turns the feature request into `.maestro.blueprint.md` plus `.maestro.blueprint.levels` (one tree level per line, defining the order of implementation slices).
2. **Ticketmaster** — For each level, the `/ticketmaster` skill (Sonnet) creates a feature branch, writes a `PRD.md` of checkbox tasks, and opens a PR. **Gate 1:** human reviews and merges.
3. **Backpressure** — `backpressure.sh` runs an agent on each merged ticket branch that writes only failing tests (no app code) for every PRD task, then pushes a `-backpressure` branch and opens a PR. **Gate 2:** human reviews and merges.
4. **Ralph implementation** — `ralph.sh <archive-folder>` iterates per PRD: read the next unchecked task → agent implements → run the task's `[test: ...]` (or `npm test`) → on pass, check the task off, append a `.agent-ledger.jsonl` entry, and update `MEMORY.md`. Capped at 10 loops per PRD. When all tasks pass, the PRD is archived to `docs/<feature-name>/PRD.<sanitized-title>.md` (collisions get a numeric suffix), `MEMORY.md` is removed, and a final commit is made. **Gate 3:** human reviews and merges.
5. **Wrap-up** — The `/summarizer` skill (Haiku) writes PR summaries, and Maestro opens a final summary PR containing its run log.

### PRD task format

```markdown
- [ ] Task description `[test: npx jest tests/foo.test.ts]`
```

The `[test: command]` annotation specifies targeted validation for that task. If omitted, `npm test` (full suite) is used. Allowed test commands must start with `npm test`, `npx jest`, `npx playwright`, `npx tsc`, or `npx biome`. PRDs are normally authored by `/ticketmaster`, not by hand.

### Key files

| File | Purpose |
|------|---------|
| `.github/scripts/maestro.sh` | Top-level orchestrator across all phases |
| `.github/scripts/ralph.sh` | Test-gated implementation loop (one PRD at a time) |
| `.github/scripts/backpressure.sh` | Failing-test generator for the active PRD |
| `.github/scripts/prompt.sh` | Claude Code CLI wrapper; handles rate-limit & error exit codes |
| `.github/scripts/fork.sh` | One-shot repo duplication helper |
| `.github/scripts/repo-slug.sh` | Parses `owner/repo` from the `origin` remote |
| `.github/prompts/ralph.md` | System prompt injected into the Ralph agent each cycle |
| `PRD.md` | Active checklist for the current branch (gitignored at the repo root) |
| `MEMORY.md` | Per-cycle agent scratchpad; overwritten each cycle, deleted when the PRD completes |
| `.agent-ledger.jsonl` | Append-only JSON-lines log of completed tasks |
| `docs/<feature>/PRD.<name>.md` | Archived PRDs after a Ralph loop completes |

The `/blueprint`, `/ticketmaster`, and `/summarizer` skills are Claude Code skills invoked inline by Maestro — they are not files in this repo.

### Files the agent must never modify

Match `.aignore`:

- `.ralph.lock`, `.ralph.log`
- Everything under `docs/` (the archive)
- `.github/scripts/maestro.sh`, `ralph.sh`, `backpressure.sh`, `prompt.sh`, `fork.sh`
- `.github/prompts/ralph.md`
- `.claude/settings.json`, `.aignore`, `biome.json`
- Test assertions or mock logic (never edit tests to force a pass)

## Writing PRDs

PRDs are normally generated by `/ticketmaster` from a blueprint, but the same guidance applies when authoring blueprints or hand-editing tickets:

- Each task should be a single, atomic unit of work
- Include a `[test: command]` annotation when possible to enable targeted backpressure
- If a task requires new dependencies, state them explicitly (e.g., "Install next, react, react-dom") — agents cannot add dependencies unless the PRD instructs it
- Order tasks so earlier tasks establish structure that later tasks build on
- Keep tasks small enough that a single agent cycle can complete them

## Conventions

- Commit messages from the loop use `chore(ai): <task description>`; backpressure commits use `chore(ai): Backpressure`; Ralph's archive commit uses `chore(ai): Archived PRD & Cleanup`
- Shell scripts use `set -euo pipefail` with explicit `set +e` around commands expected to fail (test validation, engine invocation)
- The `.aignore` file prevents AI tools from indexing the loop infrastructure — this is intentional separation between the orchestrator and the code it produces
